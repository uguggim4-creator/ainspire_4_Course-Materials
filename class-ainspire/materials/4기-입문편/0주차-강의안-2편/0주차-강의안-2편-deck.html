# 실행

다른 도구(Codex·Copilot 등)도 같은 개념을 공유합니다. Claude Code로 익히면 나머지도 빠르게 전환하실 수 있습니다.

기본 세팅 ② 기억
CLAUDE.md — 왜 중요한가
Claude는 매번 새 세션으로 시작합니다. 이전 대화를 기억하지 않습니다.
세션이 시작될 때 CLAUDE.md를 자동으로 읽고 컨텍스트를 갖춘 뒤 작업에 들어갑니다.
어떤 역할을 맡을지, 어떤 지침으로 행동할지를 적어두면 매 세션 그대로 따릅니다.
매 세션 자동으로 읽힘 — 따로 붙여넣거나 설명하지 않아도 됩니다.
역할·규칙·금지 사항을 정해둠 — Claude가 매번 같은 방식으로 일합니다.
매번 다시 설명할 필요 없음 — 한 번 작성해 두면 계속 유효합니다.
지침일 뿐 강제 사항은 아닙니다. 반드시 지키게 하려면 뒤에서 다룰 Hooks를 사용합니다.

# CLAUDE.md 예시
## 역할
강의 슬라이드 HTML을 만드는 실행팀이다.

## 규칙
- 파일은 kebab-case로 저장
- 대화체 금지. 단정형으로 작성
- CSS 클래스는 기존 것 재사용

## 금지
- _archive/ 폴더 건드리지 않기
프로젝트 폴더의 CLAUDE.md는 해당 프로젝트에만 적용됩니다. ~/.claude/CLAUDE.md는 모든 프로젝트에 공통으로 적용됩니다.
설정
README.md와는 다르다
구분 README.md CLAUDE.md
읽는 주체 사람 (팀원, 사용자, 본인) Claude (AI)
주요 내용 프로젝트 소개, 설치 방법, 사용법, 라이선스 코딩 규칙, 폴더 구조, 금지 사항, 작업 목표
언제 읽힘 사람이 GitHub에서 직접 열 때 읽습니다 Claude가 세션 시작 시 자동으로 읽습니다
형식 자유롭게 설명하는 문서 형태입니다 Claude가 따를 규칙 중심으로 간결하게 작성합니다
둘은 역할이 다릅니다. 같은 폴더에 함께 두시면 됩니다.
설정
전역 세팅 ~/.claude
settings.json이 하는 일
설정 항목설명
권한 (allow)Claude가 실행할 수 있는 명령을 영구적으로 허용합니다
권한 (deny)위험한 명령을 영구적으로 차단하여 실수를 방지합니다
환경변수 (env)API 키 같은 민감한 값을 코드 밖에서 안전하게 관리합니다
Hooks정해진 시점에 자동 실행할 명령을 등록합니다
기본 모델기본으로 사용할 Claude 모델을 지정합니다
전역 vs 프로젝트 설정. ~/.claude/settings.json은 모든 프로젝트에 공통으로 적용됩니다. 프로젝트 안의 .claude/settings.json은 해당 프로젝트에만 적용되며 우선권을 가집니다.
이전 권한 '모드'는 얼마나 자주 질문할지를 정합니다. 반면 allow/deny 규칙은 어떤 명령을 영구 허용·차단할지를 결정합니다. 이 둘은 상호 보완하며 함께 작동합니다.

/이름 형식을 통해 직접 호출하거나, 상황에 맞게 자동으로 제시됩니다.
CLAUDE.md가 너무 길어질 때 용도별로 분리해서 관리하고, 팀원들과 공유할 수 있습니다.

# 직접 만들기
사용자가 직접 만들 수도 있습니다.
~/.claude/skills/ 폴더에
마크다운 파일로 저장하면 됩니다.
자주 쓰는 배포 순서, 글쓰기 가이드, 리뷰 항목 등을 한 번 만들어두면 매번 설명할 필요가 없어 매우 편리합니다.
확장
Hooks — 정해진 순간 자동 실행
어떤 시점에 실행되나
세션 시작 시, 프롬프트 제출 전, 도구 사용 전후, 턴 종료 시점 등에 등록한 명령을 자동으로 실행합니다.
예: 작업이 끝날 때마다 자동으로 테스트를 실행하거나, 파일 저장 후 린트(Linter) 체크를 수행할 수 있습니다.
settings.json 파일에 등록하여 사용합니다.
CLAUDE.md와의 차이점. CLAUDE.md는 AI에게 주는 '지침'입니다. Claude가 참고하지만 간혹 잊을 수도 있습니다. 반면 Hooks는 AI가 아니라 프로그램(실행기)이 강제 실행하므로, 건너뛸 수 없습니다.
오류를 줄여야 하는 중요한 절차는 지침이 아니라 Hooks로 강제해야 확실하게 실행됩니다.
확장
에이전트 · 서브에이전트
서브에이전트란
규모가 큰 작업을 전문 일꾼(서브에이전트)에게 위임합니다. 이들은 독립된 공간에서 실행되며, 결과 요약만 메인에 보고합니다.
덕분에 메인 대화창의 컨텍스트가 길어지거나 지저분해지지 않습니다.
예: 검수 전담, 리서치 전담, 코드 리뷰 전담.
구조 설명. 메인 에이전트가 서브에이전트에게 하위 과업을 위임합니다. 서브에이전트는 독립적으로 작업을 수행한 뒤 요약만 반환하므로, 메인의 메모리 절약에 매우 유용합니다.
독립된 세션으로 실행되므로 토큰 소모량이 늘어납니다. 복잡하고 분리 가능한 작업에만 사용하는 것이 효율적입니다.
확장
병렬 작업
병렬이란
의존성이 없는 독립적인 작업을 동시에 처리하여 전체 대기 시간을 절약합니다.
예: 자료 검색 여러 건을 한 번에 돌리기, 영상 생성을 백그라운드로 돌리면서 다른 작업 진행.
여러 개의 서브에이전트를 한 번에 실행시켜 병렬로 처리합니다.
주의할 점. 앞 작업의 결과가 뒤 작업에 필요한 의존 관계가 있다면 병렬이 아니라 순차적으로 처리해야 합니다. 순서가 중요한 작업을 동시에 실행하면 논리적 오류가 발생할 수 있습니다.
작업 간의 의존성 여부를 판단하는 것이 중요합니다. "A가 완료되어야 B를 할 수 있다"면 순차 실행을, 그렇지 않다면 병렬 실행을 설계하세요.
참고 · 환경
필요할 때 까는 도구들
항목 무슨 용도 언제 필요

Python 데이터 처리 및 분석, AI 에이전트 구동, 다양한 자동화 스크립트를 작성하고 실행합니다. Python 코드를 짜거나 실행할 작업이 있을 때

GitHub Git 기록을 웹 저장소에 업로드하여 백업하고 팀원들과 협업할 수 있도록 돕습니다. 팀 협업, 원격 백업, 배포 연동 시
