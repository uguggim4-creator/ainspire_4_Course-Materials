# 바이브코딩 풀버전 용어 자료집

이 자료집은 바이브코딩을 하면서 자주 만나는 용어를 “작업 흐름” 기준으로 정리한 자료입니다. 단어를 사전처럼 외우기보다, 실제 프로젝트를 만들면서 어느 순간에 어떤 용어를 만나게 되는지 이해하는 데 목적이 있습니다.

## 전체 흐름

```text
아이디어 구상
↓
AI에게 작업 요청
↓
개발 환경에서 프로젝트 실행
↓
화면 만들기
↓
기능과 패키지 추가
↓
데이터 저장
↓
API와 외부 서비스 연결
↓
오류 수정
↓
GitHub에 업로드
↓
Vercel로 배포
```

## 1. AI에게 작업을 맡길 때 만나는 용어

### 바이브코딩

**도구 소개**  
바이브코딩은 사용자가 만들고 싶은 결과를 자연어로 설명하고, AI와 대화하며 코드를 작성하거나 수정하는 개발 방식입니다.

**활용 방식**  
처음부터 모든 코드를 직접 작성하기보다, 목표와 조건을 AI에게 설명하고 결과를 확인하면서 수정해 나갑니다.

**활용 예시**  
“동네 카페 예약 페이지를 만들어주세요. 모바일에서 보기 좋고, 예약 폼이 있어야 합니다.”

### 에이전트

**도구 소개**  
에이전트는 사용자의 요청을 이해하고, 필요한 도구를 사용해 작업을 수행하는 AI입니다.

**활용 방식**  
파일을 읽고, 코드를 수정하고, 테스트를 실행하는 등 실제 개발 작업을 진행할 수 있습니다.

**활용 예시**  
“로그인 페이지를 만들고, 모바일에서도 잘 보이게 수정해주세요”라고 요청하면 에이전트가 관련 파일을 찾아 작업합니다.

### 서브에이전트

**도구 소개**  
서브에이전트는 특정 역할을 맡아 작업을 도와주는 보조 에이전트입니다.

**활용 방식**  
큰 작업을 나누어 처리할 때 사용합니다. 예를 들어 한 에이전트는 오류를 분석하고, 다른 에이전트는 UI 구조를 검토할 수 있습니다.

**활용 예시**  
코드 리뷰 담당 서브에이전트가 변경된 코드에서 버그 가능성을 검토합니다.

### 컨텍스트

**도구 소개**  
컨텍스트는 AI가 현재 참고할 수 있는 정보입니다. 대화 내용, 코드 파일, 프로젝트 구조, 사용자의 요청이 포함됩니다.

**활용 방식**  
AI가 정확히 작업하려면 필요한 정보가 컨텍스트에 들어 있어야 합니다. 컨텍스트가 부족하면 AI가 잘못 추측할 수 있습니다.

**활용 예시**  
“기존 디자인과 같은 스타일로 관리자 페이지를 만들어주세요”라고 하면 기존 코드와 대화 내용이 컨텍스트로 사용됩니다.

### 프롬프트

**도구 소개**  
프롬프트는 AI에게 전달하는 작업 지시문입니다.

**활용 방식**  
무엇을 만들지, 어떤 조건을 지켜야 하는지, 결과를 어떻게 확인할지 구체적으로 작성하면 좋습니다.

**활용 예시**  
“상품 목록 페이지를 만들어주세요. 카드 형태로 보여주고, 모바일에서는 한 줄에 하나씩 보이게 해주세요.”

### 전역

**도구 소개**  
전역은 특정 파일이나 함수에만 적용되는 것이 아니라 전체 프로젝트나 넓은 범위에 적용되는 설정을 의미합니다.

**활용 방식**  
전역 스타일, 전역 설정, 전역 변수처럼 여러 곳에서 공통으로 사용하는 값이나 규칙을 관리할 때 사용합니다.

**활용 예시**  
웹사이트 전체의 글꼴과 배경색을 `globals.css`에서 설정하면 모든 페이지에 동일하게 적용됩니다.

## 2. 개발 환경을 열고 프로젝트를 실행할 때 만나는 용어

### IDE

**도구 소개**  
IDE는 코드를 작성하고 수정하는 개발 환경입니다.

**활용 방식**  
프로젝트 폴더를 열고, 파일을 수정하고, 터미널을 실행하며 개발 작업을 진행합니다.

**활용 예시**  
VS Code나 Cursor에서 프로젝트 폴더를 열고 AI에게 코드 수정을 요청합니다.

### Terminal

**도구 소개**  
Terminal은 명령어를 입력해 컴퓨터나 개발 도구를 조작하는 창입니다.

**활용 방식**  
프로젝트 실행, 패키지 설치, 빌드, Git 명령 실행 등에 사용합니다.

**활용 예시**  
터미널에서 `npm run dev`를 입력해 개발 서버를 실행합니다.

### CLI

**도구 소개**  
CLI는 Command Line Interface의 약자로, 명령어를 입력해서 프로그램을 사용하는 방식입니다.

**활용 방식**  
터미널에서 프로젝트 생성, 패키지 설치, 서버 실행, 배포 명령 등을 실행합니다.

**활용 예시**  
`npx create-next-app` 명령어로 새로운 Next.js 프로젝트를 생성합니다.

### Bash

**도구 소개**  
Bash는 터미널에서 명령어를 실행할 수 있게 해주는 쉘입니다.

**활용 방식**  
폴더 이동, 파일 확인, 프로그램 실행, 개발 서버 실행 등에 사용합니다.

**활용 예시**  
프로젝트 폴더로 이동한 뒤 `npm install`을 실행합니다.

### PowerShell

**도구 소개**  
PowerShell은 Windows에서 자주 사용하는 명령어 실행 환경입니다.

**활용 방식**  
Windows 환경에서 폴더 이동, 파일 확인, 명령 실행, 개발 서버 실행 등에 사용합니다.

**활용 예시**  
PowerShell에서 `Get-ChildItem` 명령어로 현재 폴더의 파일을 확인합니다.

### Read

**도구 소개**  
Read는 파일이나 데이터를 읽는 동작을 의미합니다.

**활용 방식**  
AI가 기존 코드 구조를 파악하거나 사용자가 파일 내용을 확인할 때 사용됩니다.

**활용 예시**  
AI에게 “현재 로그인 페이지 코드를 읽고 구조를 설명해주세요”라고 요청합니다.

### Write

**도구 소개**  
Write는 파일이나 데이터에 내용을 쓰는 동작을 의미합니다.

**활용 방식**  
새 파일을 만들거나 기존 파일을 수정할 때 사용됩니다.

**활용 예시**  
AI에게 “회원가입 페이지를 만들어주세요”라고 요청하면 필요한 파일에 코드를 작성합니다.

### NPM

**도구 소개**  
NPM은 JavaScript 패키지를 설치하고 관리하는 도구입니다.

**활용 방식**  
필요한 라이브러리를 설치하거나, `package.json`에 등록된 명령어를 실행할 때 사용합니다.

**활용 예시**  
`npm install`로 필요한 패키지를 설치하고, `npm run dev`로 프로젝트를 실행합니다.

### package.json

**도구 소개**  
`package.json`은 JavaScript 프로젝트의 설정과 패키지 정보를 담는 파일입니다.

**활용 방식**  
프로젝트 이름, 실행 명령어, 설치된 패키지 목록을 관리합니다.

**활용 예시**  
`scripts` 항목에 있는 `dev`, `build`, `start` 명령어를 NPM으로 실행합니다.

### node_modules

**도구 소개**  
`node_modules`는 설치된 JavaScript 패키지가 저장되는 폴더입니다.

**활용 방식**  
`npm install`을 실행하면 `package.json`에 적힌 패키지들이 이 폴더에 설치됩니다.

**활용 예시**  
프로젝트를 처음 받은 뒤 `npm install`을 실행하면 `node_modules` 폴더가 생성됩니다.

### Localhost

**도구 소개**  
Localhost는 내 컴퓨터에서 실행 중인 개발 서버 주소입니다.

**활용 방식**  
웹앱을 배포하기 전에 브라우저에서 미리 확인할 때 사용합니다.

**활용 예시**  
`http://localhost:3000`에 접속해 현재 개발 중인 화면을 확인합니다.

### Port

**도구 소개**  
Port는 컴퓨터 안에서 실행 중인 프로그램을 구분하는 번호입니다.

**활용 방식**  
개발 서버가 어떤 주소에서 실행되는지 구분할 때 사용합니다.

**활용 예시**  
`localhost:3000`에서 `3000`이 port 번호입니다.

## 3. 화면을 만들 때 만나는 용어

### HTML

**도구 소개**  
HTML은 웹페이지의 구조를 만드는 언어입니다.

**활용 방식**  
제목, 문단, 버튼, 이미지, 입력창 같은 화면 요소의 기본 구조를 만듭니다.

**활용 예시**  
상품 이름은 제목으로, 상품 설명은 문단으로, 구매 버튼은 버튼 요소로 구성합니다.

### CSS

**도구 소개**  
CSS는 웹페이지의 디자인을 담당하는 언어입니다.

**활용 방식**  
색상, 글꼴, 여백, 배치, 반응형 화면 등을 설정합니다.

**활용 예시**  
버튼의 배경색을 바꾸고, 모바일 화면에서는 카드가 한 줄에 하나씩 보이게 만듭니다.

### JavaScript

**도구 소개**  
JavaScript는 웹페이지에 동작을 추가하는 프로그래밍 언어입니다.

**활용 방식**  
버튼 클릭, 데이터 요청, 화면 변경 같은 기능을 구현할 때 사용합니다.

**활용 예시**  
사용자가 버튼을 클릭하면 메뉴가 열리도록 만듭니다.

### TypeScript

**도구 소개**  
TypeScript는 JavaScript에 타입 개념을 추가한 언어입니다.

**활용 방식**  
문자열, 숫자, 객체, 배열 같은 데이터의 형태를 미리 지정해 오류를 줄입니다.

**활용 예시**  
사용자 정보가 `{ name: string, age: number }` 형태여야 한다고 지정합니다.

### React

**도구 소개**  
React는 화면을 컴포넌트 단위로 만들 수 있게 해주는 JavaScript 라이브러리입니다.

**활용 방식**  
버튼, 카드, 입력창 같은 화면 요소를 재사용 가능한 컴포넌트로 나눕니다.

**활용 예시**  
`ProductCard` 컴포넌트를 만들어 여러 상품을 같은 디자인으로 보여줍니다.

### Next.js

**도구 소개**  
Next.js는 React 기반 웹앱 프레임워크입니다. 페이지 구성, 라우팅, 서버 기능, 배포 최적화 등을 제공합니다.

**활용 방식**  
React로 웹앱을 만들 때 필요한 기본 구조를 제공해 더 빠르게 개발할 수 있습니다.

**활용 예시**  
블로그, 쇼핑몰, 관리자 페이지 같은 웹앱을 Next.js로 만듭니다.

### Component

**도구 소개**  
Component는 화면을 구성하는 재사용 가능한 UI 조각입니다.

**활용 방식**  
페이지를 작은 단위로 나누어 관리하고, 같은 UI를 여러 곳에서 반복해서 사용할 수 있습니다.

**활용 예시**  
`Header`, `Button`, `LoginForm` 같은 컴포넌트를 만들어 사용합니다.

### Props

**도구 소개**  
Props는 부모 컴포넌트가 자식 컴포넌트에 전달하는 데이터입니다.

**활용 방식**  
같은 컴포넌트를 여러 곳에서 사용하면서 내용만 다르게 보여줄 때 사용합니다.

**활용 예시**  
`ProductCard`에 상품 이름과 가격을 props로 전달해 각 상품을 다르게 표시합니다.

### State

**도구 소개**  
State는 화면 안에서 변할 수 있는 상태값입니다.

**활용 방식**  
사용자 입력, 버튼 클릭, 탭 선택, 모달 열림 여부처럼 변하는 값을 관리합니다.

**활용 예시**  
검색창에 입력한 단어를 state로 저장하고, 입력값에 따라 목록을 필터링합니다.

### Routing

**도구 소개**  
Routing은 사용자가 접속한 주소에 따라 어떤 페이지를 보여줄지 정하는 방식입니다.

**활용 방식**  
Next.js에서는 파일과 폴더 구조를 기준으로 라우팅을 구성할 수 있습니다.

**활용 예시**  
`/login` 주소로 접속하면 로그인 페이지를 보여주고, `/dashboard` 주소로 접속하면 관리자 페이지를 보여줍니다.

### Page

**도구 소개**  
Page는 사용자가 특정 주소로 접속했을 때 보게 되는 화면입니다.

**활용 방식**  
Next.js에서는 페이지 파일이나 폴더 구조를 통해 화면 경로를 구성합니다.

**활용 예시**  
`/products` 페이지에서는 상품 목록을 보여주고, `/products/1` 페이지에서는 상품 상세 정보를 보여줍니다.

### Layout

**도구 소개**  
Layout은 여러 페이지에서 공통으로 사용하는 화면 구조입니다.

**활용 방식**  
헤더, 사이드바, 푸터처럼 반복되는 영역을 한 번 정의해 여러 페이지에서 공유합니다.

**활용 예시**  
관리자 페이지 전체에 같은 사이드바와 상단 메뉴가 보이도록 layout을 구성합니다.

## 4. 패키지와 라이브러리를 사용할 때 만나는 용어

### JavaScript 패키지

**도구 소개**  
JavaScript 패키지는 다른 개발자가 만들어 공개한 기능 묶음입니다.

**활용 방식**  
NPM, pnpm, Yarn 같은 패키지 관리 도구로 설치한 뒤 프로젝트에서 불러와 사용합니다.

**활용 예시**  
아이콘을 사용하려고 `lucide-react` 패키지를 설치합니다.

### Library

**도구 소개**  
Library는 특정 기능을 쉽게 구현할 수 있도록 만들어진 코드 모음입니다.

**활용 방식**  
날짜 처리, 차트, 애니메이션, 아이콘 같은 기능을 직접 만들지 않고 라이브러리를 활용합니다.

**활용 예시**  
`recharts` 라이브러리를 사용해 매출 그래프를 표시합니다.

### Framework

**도구 소개**  
Framework는 앱을 만들기 위한 기본 구조와 규칙을 제공하는 개발 도구입니다.

**활용 방식**  
라우팅, 빌드, 데이터 처리, 파일 구조 등 앱 개발에 필요한 기본 틀을 제공합니다.

**활용 예시**  
Next.js는 React 기반 프레임워크입니다.

### Dependency

**도구 소개**  
Dependency는 프로젝트 실행에 필요한 외부 패키지입니다.

**활용 방식**  
앱이 실제로 동작할 때 필요한 패키지는 dependency로 관리합니다.

**활용 예시**  
아이콘 라이브러리나 데이터 요청 라이브러리를 dependency로 설치합니다.

### DevDependency

**도구 소개**  
DevDependency는 개발 과정에서만 필요한 패키지입니다.

**활용 방식**  
테스트, 코드 검사, 포맷팅처럼 개발 중에만 필요한 도구를 관리합니다.

**활용 예시**  
코드 스타일을 검사하는 ESLint는 devDependency로 설치하는 경우가 많습니다.

## 5. API와 외부 서비스를 연결할 때 만나는 용어

### API

**도구 소개**  
API는 프로그램끼리 데이터를 주고받기 위한 연결 방식입니다.

**활용 방식**  
프론트엔드가 서버나 외부 서비스에 요청을 보내고 응답을 받아 화면에 보여줍니다.

**활용 예시**  
날씨 API에서 현재 날씨 데이터를 받아 화면에 표시합니다.

### Endpoint

**도구 소개**  
Endpoint는 API 요청을 보내는 구체적인 주소입니다.

**활용 방식**  
어떤 기능을 호출할지에 따라 다른 endpoint로 요청을 보냅니다.

**활용 예시**  
`/api/products` endpoint로 요청을 보내 상품 목록을 가져옵니다.

### Request

**도구 소개**  
Request는 클라이언트가 서버나 외부 서비스에 보내는 요청입니다.

**활용 방식**  
데이터를 가져오거나 저장하거나 수정할 때 request를 보냅니다.

**활용 예시**  
로그인 버튼을 누르면 이메일과 비밀번호를 담아 로그인 request를 보냅니다.

### Response

**도구 소개**  
Response는 request에 대해 서버가 돌려주는 응답입니다.

**활용 방식**  
요청 결과, 데이터, 오류 메시지 등을 response로 받습니다.

**활용 예시**  
상품 목록 request를 보내면 상품 데이터가 response로 돌아옵니다.

### HTTP

**도구 소개**  
HTTP는 웹에서 데이터를 주고받기 위한 기본 통신 규칙입니다.

**활용 방식**  
브라우저, 프론트엔드, 백엔드, 외부 API가 서로 통신할 때 사용합니다.

**활용 예시**  
웹페이지가 API 서버에 HTTP 요청을 보내 데이터를 가져옵니다.

### GET

**도구 소개**  
GET은 데이터를 조회할 때 사용하는 HTTP 방식입니다.

**활용 방식**  
목록이나 상세 정보를 가져올 때 사용합니다.

**활용 예시**  
상품 목록을 불러오기 위해 GET 요청을 보냅니다.

### POST

**도구 소개**  
POST는 새로운 데이터를 생성하거나 제출할 때 사용하는 HTTP 방식입니다.

**활용 방식**  
회원가입, 로그인, 글 작성, 주문 생성 등에 사용합니다.

**활용 예시**  
게시글 작성 폼을 제출하면 POST 요청으로 글 데이터를 보냅니다.

### JSON

**도구 소개**  
JSON은 데이터를 주고받을 때 자주 사용하는 형식입니다.

**활용 방식**  
프론트엔드와 백엔드가 객체 형태의 데이터를 주고받을 때 사용합니다.

**활용 예시**  
`{ "name": "홍길동", "email": "hong@example.com" }` 같은 형태로 사용자 데이터를 전달합니다.

### API Key

**도구 소개**  
API Key는 외부 서비스를 사용할 때 발급받는 인증용 키입니다.

**활용 방식**  
서비스가 요청을 보낸 사용자를 확인하고 사용 권한을 판단할 때 사용합니다.

**활용 예시**  
날씨 API나 AI API를 사용할 때 발급받은 API Key를 환경변수에 저장합니다.

### SDK

**도구 소개**  
SDK는 특정 서비스를 더 쉽게 사용할 수 있도록 제공되는 개발 도구 모음입니다.

**활용 방식**  
API를 직접 호출하지 않고 서비스에서 제공하는 함수나 라이브러리를 사용해 기능을 구현합니다.

**활용 예시**  
Supabase SDK를 사용해 로그인과 데이터 조회 기능을 구현합니다.

## 6. 데이터베이스와 백엔드를 다룰 때 만나는 용어

### Database

**도구 소개**  
Database는 앱에서 사용하는 데이터를 저장하는 공간입니다.

**활용 방식**  
회원 정보, 게시글, 상품, 주문 내역 등을 저장하고 필요할 때 다시 불러옵니다.

**활용 예시**  
게시판 앱에서 작성된 글 목록을 데이터베이스에 저장합니다.

### PostgreSQL

**도구 소개**  
PostgreSQL은 널리 사용되는 관계형 데이터베이스입니다.

**활용 방식**  
테이블, 행, 열 구조로 데이터를 저장하고 SQL로 데이터를 다룹니다.

**활용 예시**  
Neon과 Supabase는 PostgreSQL 기반 데이터베이스를 제공합니다.

### Schema

**도구 소개**  
Schema는 데이터가 어떤 구조로 저장되어야 하는지 정리한 설계도입니다.

**활용 방식**  
어떤 테이블이 필요하고, 각 데이터가 어떤 항목을 가져야 하는지 정의합니다.

**활용 예시**  
회원 schema에 `name`, `email`, `createdAt` 같은 항목을 정의합니다.

### Table

**도구 소개**  
Table은 데이터베이스에서 데이터를 저장하는 기본 단위입니다.

**활용 방식**  
사용자, 게시글, 주문, 상품처럼 종류가 다른 데이터를 각각의 테이블로 나누어 저장합니다.

**활용 예시**  
회원 정보는 `User` 테이블에 저장하고, 게시글 정보는 `Post` 테이블에 저장합니다.

### Column

**도구 소개**  
Column은 데이터베이스 테이블에서 세로 방향의 항목입니다.

**활용 방식**  
이름, 이메일, 생성일, 가격처럼 데이터의 속성을 표현합니다.

**활용 예시**  
`User` 테이블에 `id`, `email`, `name`, `createdAt` 컬럼을 만듭니다.

### Row

**도구 소개**  
Row는 데이터베이스 테이블에 저장된 실제 데이터 한 줄입니다.

**활용 방식**  
회원 한 명, 상품 하나, 게시글 하나처럼 하나의 개별 데이터를 나타냅니다.

**활용 예시**  
`User` 테이블에 저장된 “홍길동, hong@example.com”이라는 회원 정보 한 건이 하나의 row입니다.

### Relation

**도구 소개**  
Relation은 데이터베이스 테이블 사이의 관계입니다.

**활용 방식**  
사용자와 게시글, 주문과 상품처럼 서로 연결된 데이터를 표현할 때 사용합니다.

**활용 예시**  
한 명의 사용자가 여러 개의 게시글을 작성할 수 있도록 `User`와 `Post`를 연결합니다.

### ORM

**도구 소개**  
ORM은 코드에서 데이터베이스를 더 쉽게 다룰 수 있게 해주는 도구입니다.

**활용 방식**  
SQL을 직접 많이 작성하지 않고도 코드 문법으로 데이터를 생성, 조회, 수정, 삭제할 수 있습니다.

**활용 예시**  
`prisma.user.findMany()` 같은 코드로 사용자 목록을 가져올 수 있습니다.

### Prisma

**도구 소개**  
Prisma는 코드에서 데이터베이스를 쉽게 다룰 수 있게 해주는 ORM 도구입니다.

**활용 방식**  
데이터 구조를 `schema.prisma` 파일에 정의하고, Prisma Client를 통해 코드에서 데이터베이스에 접근합니다.

**활용 예시**  
`User` 모델을 만들고 Prisma로 새 사용자를 생성하거나 특정 사용자의 정보를 조회합니다.

### Migration

**도구 소개**  
Migration은 데이터베이스 구조 변경 이력을 관리하는 과정입니다.

**활용 방식**  
테이블 추가, 컬럼 변경, 관계 수정 같은 데이터베이스 구조 변경을 기록하고 적용합니다.

**활용 예시**  
사용자 테이블에 `phoneNumber` 컬럼을 추가하는 migration을 생성합니다.

### Neon

**도구 소개**  
Neon은 클라우드 기반 PostgreSQL 데이터베이스 서비스입니다.

**활용 방식**  
서버를 직접 관리하지 않고 데이터베이스를 만들고, 연결 문자열을 프로젝트에 등록해 사용합니다.

**활용 예시**  
쇼핑몰 프로젝트에서 주문 정보를 저장하기 위해 Neon 데이터베이스를 생성합니다.

### Supabase

**도구 소개**  
Supabase는 데이터베이스, 로그인, 파일 저장 기능을 제공하는 백엔드 플랫폼입니다.

**활용 방식**  
서버를 직접 만들지 않고도 로그인, 데이터 저장, 이미지 업로드 기능을 구현할 수 있습니다.

**활용 예시**  
회원가입 기능과 게시글 저장 기능을 Supabase로 구현합니다.

## 7. 로그인, 권한, 보안을 다룰 때 만나는 용어

### Auth

**도구 소개**  
Auth는 Authentication의 줄임말로, 사용자가 누구인지 확인하는 인증 기능입니다.

**활용 방식**  
회원가입, 로그인, 로그아웃, 비밀번호 재설정 같은 기능에 사용합니다.

**활용 예시**  
Supabase Auth를 사용해 이메일 로그인 기능을 구현합니다.

### Session

**도구 소개**  
Session은 사용자가 로그인한 상태를 유지하기 위한 정보입니다.

**활용 방식**  
사용자가 페이지를 이동해도 로그인 상태를 유지할 때 사용합니다.

**활용 예시**  
로그인 후 새로고침해도 사용자가 계속 로그인된 상태로 남아 있습니다.

### Token

**도구 소개**  
Token은 사용자의 인증 상태나 권한을 확인하는 데 사용하는 문자열입니다.

**활용 방식**  
API 요청을 보낼 때 token을 함께 보내 사용자가 권한을 가진 사람인지 확인합니다.

**활용 예시**  
로그인한 사용자만 게시글을 작성할 수 있도록 요청에 token을 포함합니다.

### Environment Variables

**도구 소개**  
Environment Variables는 API 키, 데이터베이스 주소 같은 민감한 값을 따로 관리하는 방식입니다.

**활용 방식**  
`.env` 파일에 값을 저장하고 코드에서는 변수 이름으로 불러와 사용합니다.

**활용 예시**  
Supabase API 키를 `.env.local`에 저장하고 로그인 기능에서 사용합니다.

### .env

**도구 소개**  
`.env`는 환경변수를 저장하는 파일입니다.

**활용 방식**  
API Key, 데이터베이스 URL, 비밀 키처럼 코드에 직접 적으면 안 되는 값을 저장합니다.

**활용 예시**  
`DATABASE_URL` 값을 `.env` 파일에 저장하고 Prisma에서 사용합니다.

### Secret Key

**도구 소개**  
Secret Key는 외부에 공개되면 안 되는 중요한 인증 키입니다.

**활용 방식**  
서버나 배포 환경의 환경변수로 안전하게 관리해야 합니다.

**활용 예시**  
결제 서비스의 secret key는 클라이언트 코드에 직접 넣지 않고 서버 환경변수로 관리합니다.

### RLS

**도구 소개**  
RLS는 Row Level Security의 줄임말로, 데이터베이스의 각 row에 접근 권한을 설정하는 기능입니다.

**활용 방식**  
사용자가 자기 데이터만 조회하거나 수정할 수 있도록 제한할 때 사용합니다.

**활용 예시**  
Supabase에서 사용자가 본인이 작성한 게시글만 수정할 수 있도록 RLS 정책을 설정합니다.

## 8. GitHub에 올리고 배포할 때 만나는 용어

### Git

**도구 소개**  
Git은 코드 변경 이력을 관리하는 버전 관리 도구입니다.

**활용 방식**  
변경 내용을 저장하고, 이전 상태와 비교하거나 되돌릴 수 있습니다.

**활용 예시**  
로그인 기능을 만든 뒤 변경 내용을 commit으로 저장합니다.

### GitHub

**도구 소개**  
GitHub는 Git으로 관리하는 코드를 온라인에 저장하고 공유하는 플랫폼입니다.

**활용 방식**  
로컬 프로젝트를 업로드하고, 협업하거나 배포 서비스와 연결할 수 있습니다.

**활용 예시**  
프로젝트 코드를 GitHub에 올린 뒤 Vercel과 연결합니다.

### Repository

**도구 소개**  
Repository는 프로젝트 코드와 변경 이력을 저장하는 공간입니다.

**활용 방식**  
GitHub에서 프로젝트별로 repository를 만들고 코드를 관리합니다.

**활용 예시**  
포트폴리오 사이트 프로젝트를 `my-portfolio` repository에 저장합니다.

### Commit

**도구 소개**  
Commit은 코드 변경 내용을 저장하는 단위입니다.

**활용 방식**  
작업이 어느 정도 완료되었을 때 변경 내용을 기록합니다.

**활용 예시**  
“로그인 페이지 추가”라는 메시지로 commit을 남깁니다.

### Branch

**도구 소개**  
Branch는 기존 코드에서 갈라져 나온 별도의 작업 공간입니다.

**활용 방식**  
새 기능을 만들거나 실험할 때 메인 코드에 바로 영향을 주지 않고 작업할 수 있습니다.

**활용 예시**  
`login-feature` 브랜치를 만들어 로그인 기능을 개발합니다.

### Push

**도구 소개**  
Push는 로컬 컴퓨터의 commit을 GitHub 같은 원격 저장소에 올리는 동작입니다.

**활용 방식**  
작업한 코드를 온라인 저장소에 반영할 때 사용합니다.

**활용 예시**  
로그인 기능 commit을 GitHub repository에 push합니다.

### Pull

**도구 소개**  
Pull은 원격 저장소의 최신 코드를 내 컴퓨터로 가져오는 동작입니다.

**활용 방식**  
다른 사람이 올린 변경 사항을 내 로컬 프로젝트에 반영할 때 사용합니다.

**활용 예시**  
팀원이 수정한 코드를 GitHub에서 pull로 가져옵니다.

### Pull Request

**도구 소개**  
Pull Request는 변경한 코드를 메인 코드에 합치기 전에 검토를 요청하는 과정입니다.

**활용 방식**  
변경 내용을 확인하고 리뷰한 뒤 병합할 수 있습니다.

**활용 예시**  
로그인 기능 브랜치에서 메인 브랜치로 Pull Request를 생성합니다.

### Vercel

**도구 소개**  
Vercel은 웹사이트와 웹앱을 쉽게 배포할 수 있는 호스팅 플랫폼입니다.

**활용 방식**  
GitHub 저장소와 연결하면 코드가 변경될 때 자동으로 빌드와 배포가 진행됩니다.

**활용 예시**  
Next.js로 만든 사이트를 Vercel에 배포해 인터넷 주소로 접속할 수 있게 만듭니다.

### Build

**도구 소개**  
Build는 개발한 코드를 실제 배포 가능한 형태로 변환하는 과정입니다.

**활용 방식**  
배포 전에 코드 오류나 설정 문제를 확인하는 데 사용됩니다.

**활용 예시**  
`npm run build`를 실행해 Vercel 배포 전에 오류가 없는지 확인합니다.

### Deploy

**도구 소개**  
Deploy는 만든 웹사이트나 앱을 인터넷에서 접속할 수 있도록 배포하는 과정입니다.

**활용 방식**  
Vercel, Netlify, AWS 같은 플랫폼에 코드를 올려 실제 서비스 주소를 만듭니다.

**활용 예시**  
GitHub에 코드를 push하면 Vercel이 자동으로 새 버전을 배포합니다.

### Production

**도구 소개**  
Production은 실제 사용자가 접속하는 운영 환경입니다.

**활용 방식**  
테스트용 환경과 구분해 안정적으로 관리해야 합니다.

**활용 예시**  
`myservice.com`에 접속하는 실제 서비스 화면이 production 환경입니다.

### Preview

**도구 소개**  
Preview는 실제 배포 전에 변경 사항을 미리 확인하는 환경입니다.

**활용 방식**  
Pull Request나 브랜치별로 미리보기 배포를 만들어 검토할 수 있습니다.

**활용 예시**  
Vercel이 Pull Request마다 preview URL을 생성해 변경된 화면을 확인할 수 있게 합니다.

### Domain

**도구 소개**  
Domain은 사용자가 웹사이트에 접속할 때 입력하는 주소입니다.

**활용 방식**  
배포한 사이트에 보기 쉬운 주소를 연결할 때 사용합니다.

**활용 예시**  
`example.com` 도메인을 Vercel 프로젝트에 연결합니다.

### DNS

**도구 소개**  
DNS는 도메인을 실제 서버 주소와 연결해주는 시스템입니다.

**활용 방식**  
도메인을 Vercel 같은 배포 서비스에 연결할 때 DNS 설정을 수정합니다.

**활용 예시**  
도메인 관리 페이지에서 Vercel이 안내한 DNS 레코드를 추가합니다.

## 9. 오류를 읽고 수정할 때 만나는 용어

### Error

**도구 소개**  
Error는 코드나 설정 문제로 프로그램이 정상적으로 실행되지 않는 상태입니다.

**활용 방식**  
오류 메시지를 읽고 원인이 되는 파일과 코드를 찾아 수정합니다.

**활용 예시**  
필요한 패키지가 설치되지 않아 앱 실행 중 error가 발생합니다.

### Log

**도구 소개**  
Log는 프로그램 실행 중 발생한 정보나 오류 기록입니다.

**활용 방식**  
오류 원인을 찾거나 프로그램 흐름을 확인할 때 사용합니다.

**활용 예시**  
터미널 로그를 AI에게 전달해 오류 원인을 분석하게 합니다.

### Stack Trace

**도구 소개**  
Stack Trace는 오류가 발생한 코드의 호출 흐름을 보여주는 정보입니다.

**활용 방식**  
어떤 파일과 함수에서 오류가 시작되었는지 추적할 때 사용합니다.

**활용 예시**  
오류 메시지에 표시된 파일명과 줄 번호를 보고 문제 위치를 찾습니다.

### Build Error

**도구 소개**  
Build Error는 배포 가능한 형태로 변환하는 과정에서 발생하는 오류입니다.

**활용 방식**  
타입 오류, 잘못된 import, 환경변수 누락, 설정 문제 등을 확인합니다.

**활용 예시**  
Vercel 배포 중 build error가 발생해 배포가 실패합니다.

### Runtime Error

**도구 소개**  
Runtime Error는 앱이 실행되는 중에 발생하는 오류입니다.

**활용 방식**  
화면 조작, API 요청, 데이터 처리 과정에서 발생한 문제를 확인합니다.

**활용 예시**  
페이지는 열리지만 버튼을 누르면 runtime error가 발생합니다.

### Type Error

**도구 소개**  
Type Error는 데이터의 형태가 예상과 다를 때 발생하는 오류입니다.

**활용 방식**  
TypeScript 프로젝트에서 잘못된 데이터 사용을 미리 찾는 데 도움이 됩니다.

**활용 예시**  
숫자를 받아야 하는 함수에 문자열을 전달하면 type error가 발생할 수 있습니다.

### Lint

**도구 소개**  
Lint는 코드 스타일과 잠재적인 문제를 검사하는 과정입니다.

**활용 방식**  
사용하지 않는 변수, 잘못된 문법, 스타일 문제 등을 확인합니다.

**활용 예시**  
`npm run lint`를 실행해 코드 품질 문제를 확인합니다.

### Test

**도구 소개**  
Test는 코드가 의도한 대로 동작하는지 확인하는 과정입니다.

**활용 방식**  
기능 수정 후 기존 기능이 깨지지 않았는지 확인합니다.

**활용 예시**  
로그인 기능을 수정한 뒤 올바른 계정으로 로그인되는지 테스트합니다.

### Diff

**도구 소개**  
Diff는 코드가 이전과 비교해 어떻게 바뀌었는지 보여주는 변경 내역입니다.

**활용 방식**  
AI가 수정한 코드나 개발자가 변경한 내용을 검토할 때 사용합니다.

**활용 예시**  
AI에게 “수정 전후 diff를 보여주세요”라고 요청해 변경된 부분만 확인합니다.

### Patch

**도구 소개**  
Patch는 기존 코드에 적용할 수 있는 변경 묶음입니다.

**활용 방식**  
코드 전체를 다시 작성하지 않고 필요한 부분만 바꿀 때 사용합니다.

**활용 예시**  
버튼 색상과 텍스트만 수정하는 patch를 적용해 UI를 빠르게 변경합니다.

### Refactor

**도구 소개**  
Refactor는 기능은 그대로 유지하면서 코드 구조를 더 읽기 쉽고 관리하기 좋게 개선하는 작업입니다.

**활용 방식**  
중복 코드를 줄이거나, 복잡한 함수를 나누거나, 변수명을 명확하게 바꿀 때 사용합니다.

**활용 예시**  
하나의 긴 컴포넌트를 여러 개의 작은 컴포넌트로 나누어 유지보수를 쉽게 만듭니다.

## 실전 요청 예시

### 오류 수정 요청

**나쁜 요청**

```text
오류 고쳐주세요.
```

**좋은 요청**

```text
npm run build를 실행했더니 아래 오류가 발생했습니다.
오류 원인을 설명하고, 관련 파일을 찾아 수정해주세요.
수정 후 어떤 부분이 바뀌었는지도 알려주세요.
```

### UI 수정 요청

**나쁜 요청**

```text
예쁘게 만들어주세요.
```

**좋은 요청**

```text
메인 페이지의 상품 카드 UI를 정리해주세요.
PC에서는 3열, 모바일에서는 1열로 보이게 하고,
기존 색상과 버튼 스타일은 유지해주세요.
```

### 기능 추가 요청

**나쁜 요청**

```text
로그인 만들어주세요.
```

**좋은 요청**

```text
Supabase Auth를 사용해서 이메일 로그인 기능을 추가해주세요.
로그인, 로그아웃, 로그인 상태 유지가 가능해야 합니다.
필요한 환경변수 이름도 함께 정리해주세요.
```

### 배포 오류 요청

**나쁜 요청**

```text
배포가 안 됩니다.
```

**좋은 요청**

```text
Vercel 배포 중 build error가 발생했습니다.
아래 로그를 보고 원인을 설명한 뒤, 필요한 파일을 수정해주세요.
로컬에서 확인해야 할 명령어도 알려주세요.
```
