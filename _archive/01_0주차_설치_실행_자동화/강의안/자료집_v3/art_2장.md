## 2. 꼭 알아야 할 핵심 단어 — 층위별 정리

로컬 AI를 쓰기 시작하면 비슷해 보이는 단어가 한꺼번에 등장합니다. 이때 중요한 기준은 단어를 외우는 것이 아니라, **같은 층위의 단어끼리 묶어 보는 것**입니다. API와 MCP는 연결 규칙의 말이고, 토큰과 컨텍스트는 AI가 정보를 처리하는 방식의 말이며, bash와 command는 컴퓨터에 명령을 내리는 방식의 말입니다.

로컬 AI가 파일을 보거나 명령을 실행하는 경우는 언제나 조건이 붙습니다. **그런 기능이 있는 도구를 사용하고, 사용자가 권한을 부여한 경우**에만 가능합니다. 단어의 층위를 나누면 “AI가 생각하는 것”, “AI가 외부와 연결되는 것”, “컴퓨터가 실행하는 것”을 훨씬 덜 헷갈리게 볼 수 있습니다.

## 2.1 층위 1 · AI와 외부를 잇는 것들

이 층위의 단어들은 AI가 자기 바깥의 파일, 앱, 웹서비스, 작업 절차와 만나는 통로를 설명합니다. 연결 규칙, 연결 문서, 실행 권한을 다루는 말들이며, AI의 내부 처리 방식이나 개발 언어 이름과는 다른 층위입니다.

### CLI

CLI(Command Line Interface)는 버튼 대신 글자 명령으로 프로그램을 조작하는 방식입니다. **마우스로 클릭하는 앱 화면이 아니라, 터미널에 명령을 입력해 프로그램을 다루는 방식**이라고 보면 됩니다. 로컬 AI 도구나 영상 변환 도구는 CLI 형태로 설치되거나 실행되는 경우가 많습니다.

CLI 자체가 권한을 뜻하지는 않습니다. 실제로 무엇을 할 수 있는지는 실행되는 프로그램과 사용자가 부여한 권한에 따라 달라집니다. 비유하면, 앱 화면의 버튼을 누르는 대신 글자로 “이 버튼을 눌러 주세요”라고 지시하는 방식에 가깝습니다.

영상 작업에서는 이런 형태로 자주 마주칩니다:

- 로컬 AI 도구 실행
- 영상 파일 변환
- 자막 파일 생성
- `tool-name input.mp4` 같은 명령 실행

비슷한 말과 나란히 놓으면 구분이 쉬워집니다. **터미널은 명령을 입력하는 창이고, Shell은 명령을 해석하는 프로그램이며, command는 실제 입력 문장입니다.** CLI는 이 모든 것을 포함해 글자 명령으로 프로그램을 조작하는 방식 전체를 뜻합니다.

### API

API(Application Programming Interface)는 프로그램끼리 기능이나 데이터를 주고받기 위해 정한 약속입니다. **한 프로그램이 다른 프로그램에게 “이 데이터를 주세요”, “이 작업을 해 주세요”라고 요청할 때 따르는 규칙**입니다.

예를 들어 영상 플랫폼의 데이터, 캘린더, 클라우드 저장소, 번역 서비스 등을 자동화에 연결할 때 API를 만날 수 있습니다. API를 쓰려면 API 키나 인증 토큰처럼 권한을 증명하는 값이 필요할 수 있습니다. 사람끼리 전화번호와 통화 예절이 있어야 연락할 수 있듯이, 프로그램끼리도 약속된 연락 방법이 필요하고 그 약속이 API입니다.

크리에이터 작업에서는 이런 곳에 쓰입니다:

- 영상 업로드 자동화
- 댓글 수집
- 조회수 데이터 가져오기
- 자막 번역 요청
- 썸네일 생성 서비스 호출

API와 MCP는 자주 함께 보이지만 같은 말은 아닙니다. **API는 프로그램끼리 주고받는 넓은 약속이고, MCP는 AI 앱과 외부 도구를 잇기 위해 표준화한 방식**입니다. MCP가 API를 대체하는 것이 아니라, MCP 서버가 뒤에서 기존 API를 호출할 수도 있습니다. 또 API의 인증 토큰은 로그인·권한을 위한 값이고, AI 토큰은 글자 조각 처리 단위라서 전혀 다른 뜻입니다.

### MCP

MCP(Model Context Protocol)는 AI 앱이 외부 도구와 데이터를 다루기 쉽게 연결하기 위한 표준화된 프로토콜입니다. 여기서 프로토콜은 **서로 데이터를 주고받기 위해 정한 공통 규칙**에 가깝습니다.

MCP는 AI 앱이 파일, 데이터베이스, 브라우저, 업무 도구 같은 외부 자원과 연결될 때 공통된 방식으로 주고받게 해줍니다. MCP 서버는 꼭 인터넷 어딘가에 있는 서버만 뜻하지 않습니다. 내 컴퓨터에서 실행되는 로컬 프로그램일 수도 있고, 외부 서비스와 연결된 서버일 수도 있습니다.

다만 MCP는 연결 방식을 표준화하는 것이지, 안전을 자동으로 보장하는 장치는 아닙니다. **실제로 어디까지 동작할 수 있는지는 권한 설정과 AI 앱의 구현에 따라 달라집니다.** 비유하면 여러 외부 도구를 AI가 알아듣기 쉬운 공통 플러그 모양으로 꽂게 해주는 연결 규격에 가깝습니다.

로컬 AI 앱에서는 이런 표현으로 자주 보게 됩니다:

- “MCP server 추가”
- “filesystem MCP”
- “browser MCP”
- “database MCP”
- AI가 권한을 부여받은 폴더의 파일을 읽는 설정
- AI가 작업 도구의 데이터를 가져오게 하는 설정

API, Skill, 권한과도 구분해 두면 좋습니다. API는 프로그램끼리 주고받는 넓은 약속이고, MCP는 그중에서도 AI 앱이 외부 도구를 다루기 쉽도록 만든 표준화된 연결 방식입니다. Skill은 작업 절차와 기준이고, MCP는 외부 도구와 연결하는 방식입니다. **MCP를 쓴다고 해서 자동으로 안전한 것은 아니므로, 어떤 권한을 허용했는지 확인해야 합니다.**

### Skill

Skill은 AI가 특정 작업을 할 때 따를 절차, 기준, 예시를 묶어 둔 작업 지침입니다. **“이런 순서로 작업하세요”, “이 기준으로 판단하세요”, “이 형식으로 결과를 내세요” 같은 내용을 담는 업무 매뉴얼**에 가깝습니다.

Skill 자체는 권한이 아닙니다. 실제로 파일을 읽거나 명령을 실행하거나 API를 호출하는 일은 별도의 도구 권한이 있을 때만 가능합니다. 직원에게 “우리 채널 자막은 이런 규칙으로 만들고, 썸네일 문구는 이런 기준으로 고르세요”라고 적어 둔 문서를 주는 것과 비슷합니다.

반복되는 제작 기준을 AI에게 적용할 때 Skill을 보게 됩니다:

- 쇼츠 자막 스타일 규칙
- 영상 제목 후보 생성 기준
- 썸네일 문구 평가표
- 업로드 설명문 작성 순서

Skill은 절차와 기준입니다. MCP나 API는 외부 도구와 연결하는 방식이고, 권한은 AI가 실제 파일이나 도구에 접근할 수 있게 허용하는 설정입니다. **Skill이 있다고 해서 AI가 자동으로 파일을 열거나 명령을 실행할 수 있는 것은 아닙니다.**

### .md, Markdown

`.md`는 Markdown 문서 파일입니다. Markdown은 제목·목록·강조 같은 문서 구조를 `# 제목`, `- 목록`, `**강조**`처럼 간단한 기호로 표현하는 형식입니다. **워드 문서보다 가볍고, 메모장보다 구조가 잘 보이는 설명서 형식**이라고 생각하면 됩니다.

AI 도구는 프로젝트 안의 `CLAUDE.md`, `AGENTS.md`, `README.md` 같은 Markdown 파일을 지시문이나 설명서로 읽는 경우가 있습니다. 다만 어떤 파일을 자동으로 읽는지는 도구마다 다르며, 파일 접근은 권한과 앱 구현에 따라 달라집니다.

영상 프로젝트에서도 Markdown은 꽤 유용합니다. 예를 들어 로컬 AI에게 “이 폴더에서는 이런 규칙으로 작업하세요”라고 알려주는 지시문 파일로 쓸 수 있고, 자막 규칙, 파일명 규칙, 업로드 체크리스트를 적어 둘 때도 편리합니다.

Markdown 파일은 문서입니다. 보통 그 자체가 실행되는 프로그램은 아닙니다. 스크립트는 컴퓨터가 순서대로 실행하는 명령 묶음이고, Skill의 내용이 `.md` 파일로 적혀 있을 수는 있지만 **`.md` 자체가 곧 Skill인 것은 아닙니다.**

실제로 AI 도구에 주는 지시문 `.md`는 이렇게 생겼습니다:

```markdown
# 이 폴더 작업 규칙 (AGENTS.md)

## 자막 규칙
- 한 줄에 16자 이내, 두 줄까지만
- 외래어는 한글 표기 우선

## 파일명 규칙
- YYYY-MM-DD_주제_v01.확장자

## 업로드 전 체크
- 개인정보·저작권 이미지 확인
```
*`#`은 제목, `-`는 목록을 뜻합니다. 사람이 읽기 쉬운 그대로 AI도 지시문으로 읽습니다.*

### 도구 권한

도구 권한은 AI나 프로그램이 파일, 명령어, 외부 서비스에 접근해도 되는지를 정하는 허용 범위입니다. **로컬 AI가 파일을 읽거나 명령을 실행하거나 API를 호출하려면, 해당 기능을 제공하는 도구와 사용자의 권한 허용이 필요합니다.**

어떤 도구는 매번 승인하게 하고, 어떤 도구는 특정 폴더나 특정 명령만 허용하게 할 수 있습니다. 권한 구조는 AI 앱마다 다를 수 있으므로 공식 문서를 확인해야 합니다. 비유하면 직원에게 사무실 열쇠, 창고 열쇠, 회사 계정 비밀번호를 어디까지 줄지 정하는 것과 비슷합니다.

실제로는 이런 승인 화면에서 자주 보게 됩니다:

- “이 명령을 실행해도 될까요?”
- “이 폴더를 읽어도 될까요?”
- “이 MCP 서버를 허용할까요?”

Skill은 일을 하는 절차이고, 권한은 실제로 접근하거나 실행해도 되는 범위입니다. MCP는 연결 방식이며, 그 연결이 안전한지는 권한 설정과 구현에 달려 있습니다. **AI가 무엇을 할 수 있는지는 ‘지침’보다 ‘허용된 권한’에 더 직접적으로 좌우됩니다.**

## 2.2 층위 2 · AI가 머리 쓰는 방식

이 층위의 단어들은 AI가 정보를 담고, 계산하고, 작업을 나누는 방식에 관한 말입니다. 외부 서비스와 연결하는 API/MCP나, 컴퓨터에 직접 입력하는 명령어와는 다른 층위입니다.

### 컨텍스트, Context

컨텍스트는 AI가 답변하거나 작업할 때 참고할 수 있는 현재 정보 묶음입니다. **AI의 책장 전체가 아니라, 지금 작업 책상 위에 펼쳐진 서류 더미**라고 보면 이해하기 쉽습니다.

컨텍스트에는 대화 내용, 사용자가 붙여 넣은 자료, 도구가 읽어 온 파일 내용, 작업 지시문 등이 포함될 수 있습니다. 다만 AI가 내 컴퓨터의 모든 파일을 항상 아는 것은 아닙니다. 권한을 부여받고, 도구가 해당 정보를 불러와 컨텍스트에 넣었을 때 참고할 수 있습니다.

이 단어는 이런 표현으로 자주 보게 됩니다:

- “컨텍스트가 길다”
- “컨텍스트에 파일을 추가한다”
- “컨텍스트 윈도우가 크다”

긴 대본, 여러 개의 자막 파일, 브랜드 가이드, 이전 업로드 기록을 AI가 참고하게 할 때 컨텍스트가 중요해집니다. 파일은 컴퓨터에 저장된 원본이고, 컨텍스트는 AI가 현재 참고하도록 받은 정보입니다. **메모리는 도구에 따라 장기적으로 저장되는 사용자 정보 기능을 뜻할 수 있어, 컨텍스트와 항상 같은 말은 아닙니다.**

### 토큰, Token

여기서 말하는 토큰은 AI가 글을 처리할 때 사용하는 작은 단위, 즉 글자 조각입니다. **AI는 문장을 사람처럼 정확히 한 글자씩 또는 한 단어씩만 세지 않고, 토큰이라는 조각 단위로 나누어 처리합니다.**

한국어, 영어, 숫자, 기호에 따라 토큰으로 잘리는 방식은 달라질 수 있습니다. 컨텍스트 크기, 처리 속도, 사용량 계산에서 토큰이라는 말을 자주 보게 됩니다. 비유하면 AI가 글을 씹어 삼키기 좋게 잘라 놓은 작은 조각입니다.

이런 표현에서 자주 등장합니다:

- “토큰 한도 초과”
- “200k context”
- “입력 토큰”
- “출력 토큰”
- “사용량 계산”

긴 대본이나 자막 파일을 한 번에 넣을 수 있는지 판단할 때 토큰 개념이 중요합니다. 다만 AI 토큰과 인증 토큰은 전혀 다릅니다. **AI 토큰은 글자 조각 처리 단위이고, 인증 토큰은 로그인이나 API 사용 권한을 증명하는 비밀 문자열**입니다. API 키, 로그인 토큰, 인증 토큰은 보안 정보이고, AI 토큰은 사용량·문맥 길이를 계산하는 단위입니다.

### 에이전트, Agent

에이전트는 목표를 받아 여러 단계를 스스로 계획하고 실행하는 AI 작업 방식입니다. **챗봇이 질문과 답변 중심이라면, 에이전트는 목표를 받고 필요한 단계를 나누어 처리하려는 방식**에 가깝습니다.

예를 들어 “이 폴더의 영상 파일을 정리하고, 제목 후보를 만들고, 설명문 초안을 작성해 주세요”처럼 목표를 주면 에이전트는 필요한 일을 쪼개어 처리하려고 합니다. 다만 파일 읽기, 명령 실행, 외부 서비스 호출은 권한을 부여받은 경우에만 가능합니다.

영상 제작 흐름에서는 여러 단계가 이어지는 작업에서 자주 보게 됩니다:

- 자료 수집
- 대본 초안 작성
- 자막 정리
- 썸네일 문구 후보 생성
- 업로드 설명문 작성

챗봇과 에이전트는 완전히 끊어진 개념이라기보다 정도 차이가 있습니다. 챗봇은 대화 중심이고, 에이전트는 목표 달성을 위해 계획과 실행을 더 적극적으로 합니다. **에이전트도 권한이 없으면 실제 파일 수정이나 명령 실행은 할 수 없습니다.**

### 서브에이전트

서브에이전트는 메인 에이전트가 일부 작업을 떼어 맡기는 보조 AI 일꾼입니다. **큰 작업을 작은 역할로 나누어 처리한다는 점이 핵심**입니다.

메인 에이전트가 전체 목표를 관리하고, 서브에이전트가 조사, 검수, 요약, 코드 수정 같은 일부 작업을 맡는 구조입니다. 도구에 따라 서브에이전트가 실제로 어떻게 만들어지고 어떤 권한을 갖는지는 다를 수 있습니다.

비유하면 총괄 PD가 있고, 그 아래에 자료 조사 담당, 자막 검수 담당, 썸네일 문구 담당이 따로 붙는 구조와 비슷합니다. 이런 작업에서 서브에이전트를 보게 됩니다:

- 긴 영상 대본 검수
- 여러 자료 비교
- 자막 오류 찾기
- 업로드 체크리스트 확인

에이전트는 전체 목표를 맡는 주체이고, 서브에이전트는 그중 일부 역할을 맡는 보조 주체입니다. Skill은 절차와 기준 묶음이고, 서브에이전트는 작업을 수행하는 역할 단위입니다. **지침과 역할은 서로 다르다는 점을 구분해 두면 좋습니다.**

## 2.3 층위 3 · 컴퓨터에게 시키는 말

이 층위의 단어들은 컴퓨터에게 직접 일을 시킬 때의 실행 환경, 명령 문장, 실행 파일을 설명합니다. AI가 권한을 부여받은 경우, 이 층위의 명령을 대신 입력하거나 실행할 수 있습니다.

### 터미널, Terminal

터미널은 명령어를 입력하고 결과를 보는 창입니다. **컴퓨터와 글자로 대화하는 창**이라고 생각하면 됩니다.

다만 터미널 자체가 명령을 이해하는 주체는 아닙니다. 터미널 안에서 bash, zsh, PowerShell 같은 Shell이 명령을 해석합니다. 그래서 터미널은 장소이고, Shell은 그 안에서 명령을 읽고 처리하는 프로그램에 가깝습니다.

터미널은 이런 안내에서 자주 등장합니다:

- 로컬 AI 도구 설치
- 영상 처리 도구 실행
- 스크립트 실행
- 개발 서버 실행

터미널, Shell, command, CLI는 자주 함께 나오지만 역할이 다릅니다. **터미널은 창, Shell은 명령 해석기, command는 실제 입력 문장, CLI는 글자로 프로그램을 조작하는 방식**입니다.

### Shell

Shell은 터미널에 입력한 명령을 해석해 운영체제에 전달하는 프로그램입니다. **사람이 입력한 명령을 컴퓨터가 실행할 수 있게 통역하는 통역사**라고 볼 수 있습니다.

Shell은 사용자가 입력한 명령을 읽고, 어떤 프로그램을 실행할지 판단합니다. bash, zsh, PowerShell 등이 Shell에 해당합니다. 파일 경로, 환경 변수, 명령 연결 같은 규칙도 Shell이 처리합니다.

이 단어는 이런 표현에서 보게 됩니다:

- “shell command”
- “bash에서 실행”
- “zsh 설정”
- “PowerShell 명령”

Shell은 명령 해석기이고, bash는 Shell의 한 종류입니다. 터미널은 Shell을 보여주는 창입니다. **터미널과 Shell을 구분하면 설치 문서나 오류 메시지를 읽을 때 덜 헷갈립니다.**

### bash

bash는 Linux와 macOS 계열 환경에서 널리 쓰이는 Shell의 한 종류입니다. **컴퓨터에게 짧은 명령을 전달하는 대표적인 통역사 중 하나**라고 보면 됩니다.

설치 문서나 자동화 안내에서 `bash install.sh` 같은 명령을 자주 볼 수 있습니다. Windows에서는 PowerShell, Git Bash, WSL 같은 환경에서 bash와 비슷한 명령을 접할 수 있습니다. bash는 웹사이트를 만드는 개발 언어라기보다, 컴퓨터 명령을 해석하고 실행하는 Shell입니다.

bash는 이런 작업에서 자주 등장합니다:

- `.sh` 파일 실행
- 프로그램 설치
- 폴더 이동
- 파일 복사
- 여러 명령 이어서 실행

bash는 Shell입니다. Python은 프로그래밍 언어이고, Node는 JavaScript 실행 환경이며, PowerShell은 Windows에서 많이 쓰이는 다른 종류의 Shell입니다. **bash를 개발 언어로 보기보다 명령 해석기로 이해하면 흐름이 명확해집니다.**

### command

command는 터미널에 입력하는 실제 명령 문장입니다. **컴퓨터에게 보내는 한 줄짜리 작업 지시문**이라고 볼 수 있습니다.

command는 보통 프로그램 이름, 옵션, 대상 파일로 구성됩니다. 예를 들어 `node script.mjs input.mp4`에는 무엇을 실행할지와 어떤 파일을 대상으로 할지가 함께 들어갑니다. 하나의 command를 직접 입력할 수도 있고, 여러 command를 스크립트 파일에 모아 둘 수도 있습니다.

이 단어는 AI가 “다음 명령을 실행하면 됩니다”라고 안내할 때 자주 보입니다. 또는 로컬 AI가 권한을 부여받아 터미널 명령을 실행할 때도 command를 만나게 됩니다.

command는 실제 입력 문장이고, CLI는 글자로 조작하는 방식입니다. 터미널은 command를 입력하는 창이고, 스크립트는 command나 코드를 모아 둔 파일입니다. **한 줄 명령인지, 명령을 담은 파일인지 구분하는 것이 중요합니다.**

### 스크립트, script

스크립트는 컴퓨터가 순서대로 실행하도록 적어 둔 명령 묶음 파일입니다. **매번 직접 입력하기 귀찮은 여러 명령을 순서대로 적어 둔 자동 실행 목록**에 가깝습니다.

스크립트는 반복 작업을 자동화할 때 자주 쓰입니다. `.sh`는 Shell 스크립트, `.mjs`는 Node에서 실행하는 JavaScript 계열 스크립트, `.py`는 Python 스크립트인 경우가 많습니다. 스크립트가 실제로 파일을 바꾸거나 외부 서비스에 접속하는지는 실행 권한과 내용에 따라 달라집니다.

영상 작업에서는 이런 자동화에 쓰일 수 있습니다:

- 영상 파일 일괄 이름 변경
- 자막 파일 정리
- 썸네일 이미지 크기 변환
- 업로드용 설명문 생성

command는 한 줄 명령이고, 스크립트는 여러 명령이나 코드를 파일로 묶은 것입니다. Markdown 문서는 설명서에 가깝고, 일반적으로 실행 파일이 아닙니다. **문서 파일과 실행 파일을 구분해 두면 권한을 판단할 때도 도움이 됩니다.**

### 전역, global

전역(global)은 특정 폴더 하나가 아니라 컴퓨터 전체나 사용자 계정 전체에 적용된다는 뜻입니다. **한 작업방에만 붙인 규칙이 아니라, 사무실 전체에 붙인 공지문**에 가깝습니다.

프로그램을 전역으로 설치하면 여러 폴더에서 같은 명령을 사용할 수 있습니다. 설정을 전역으로 바꾸면 특정 프로젝트뿐 아니라 다른 작업에도 영향을 줄 수 있습니다. 편리하지만, 프로젝트마다 다른 버전이나 다른 설정이 필요할 때는 헷갈릴 수 있습니다.

이런 표현에서 전역을 보게 됩니다:

- `npm install -g ...`
- “global settings”
- “global config”
- “전역 지시문”

전역(global)은 전체에 적용되고, 로컬(local)은 현재 폴더나 현재 프로젝트에만 적용됩니다. **전역 설치는 편하지만, 여러 프로젝트를 다룰 때 버전 충돌이 생길 수 있습니다.**

### node

Node.js는 JavaScript를 브라우저 밖, 즉 내 컴퓨터에서 실행할 수 있게 해주는 실행 환경입니다. **웹에서 쓰던 JavaScript를 내 컴퓨터에서도 일하게 해주는 실행 엔진**이라고 생각하면 됩니다.

JavaScript는 원래 웹브라우저에서 많이 쓰이는 언어지만, Node.js를 사용하면 컴퓨터에서 직접 JavaScript 파일을 실행할 수 있습니다. 많은 AI 도구, MCP 서버, 자동화 스크립트가 Node.js 기반으로 제공되기도 합니다. 그래서 `node`, `npm`, `npx` 같은 명령을 함께 보게 되는 경우가 많습니다.

Node.js는 이런 명령에서 자주 등장합니다:

- `node server.mjs`
- `npm install`
- `npx ...`
- 로컬 MCP 서버 실행
- 웹앱 개발 서버 실행
- 자동화 스크립트 실행

JavaScript는 언어이고, Node.js는 JavaScript를 실행하는 환경입니다. npm은 Node.js 생태계의 패키지 설치 도구이고, Next.js는 웹사이트·웹앱을 만들기 위한 프레임워크입니다. **언어, 실행 환경, 설치 도구, 프레임워크를 나눠 보면 훨씬 덜 헷갈립니다.**

## 2.4 층위 4 · 개발 언어 (이름만 알면 되는 것)

이 층위의 단어들은 AI가 결과물을 만들 때 사용하는 개발 기술의 이름입니다. 비개발자는 문법을 외우기보다, **AI가 어떤 재료로 어떤 종류의 결과물을 만들고 있는지 알아보는 정도**면 충분합니다.

### Python

Python은 자동화, 데이터 처리, 파일 정리, AI 관련 작업에 널리 쓰이는 프로그래밍 언어입니다. **반복 업무를 시키기 좋은 만능 사무 도구 같은 언어**라고 볼 수 있습니다.

Python은 비교적 읽기 쉬운 문법 때문에 자동화 작업에서 자주 선택됩니다. AI가 `.py` 파일을 만들어 영상 파일 목록을 정리하거나, 자막 텍스트를 가공하거나, CSV 데이터를 분석할 수 있습니다. 직접 코딩하지 않더라도 “이 결과물은 Python 스크립트구나” 정도를 알면 흐름을 이해하기 쉽습니다.

크리에이터 작업에서는 이런 곳에서 보게 됩니다:

- 자막 정리
- 파일명 일괄 변경
- 조회수 데이터 분석
- 대본 텍스트 가공
- 이미지·영상 처리 자동화

Python은 언어입니다. 스크립트는 Python 같은 언어로 적은 실행 파일일 수 있습니다. Node.js는 JavaScript를 실행하는 환경입니다. **Python과 Node.js를 같은 종류로 보기보다, 언어와 실행 환경을 구분해 두는 편이 좋습니다.**

### JavaScript / TypeScript

JavaScript는 웹에서 많이 쓰이는 프로그래밍 언어이고, TypeScript는 JavaScript에 타입 규칙을 더한 언어입니다. **JavaScript가 기본 작업 언어라면, TypeScript는 맞춤법 검사와 안전장치를 더한 버전**이라고 볼 수 있습니다.

JavaScript는 웹페이지의 동작, 웹앱, Node.js 자동화에 널리 쓰입니다. TypeScript는 큰 프로젝트에서 실수를 줄이기 위해 변수와 데이터 형태를 더 엄격하게 확인하도록 만든 JavaScript 확장에 가깝습니다. AI가 `.js`, `.mjs`, `.ts`, `.tsx` 파일을 만들거나 수정할 때 이 이름들을 보게 됩니다.

이 기술들은 이런 결과물에 자주 쓰입니다:

- 웹사이트
- 채널 운영 대시보드
- 업로드 관리 도구
- MCP 서버
- Node.js 자동화 스크립트

JavaScript와 Java는 이름이 비슷하지만 다른 언어입니다. TypeScript는 실행될 때 보통 JavaScript로 변환됩니다. Node.js는 JavaScript를 내 컴퓨터에서 실행하게 해주는 환경입니다. **JavaScript, TypeScript, Node.js를 한 덩어리로 보되 각자의 역할은 나눠서 이해하면 좋습니다.**

### React

React는 웹 화면을 작은 부품 단위로 만들기 위한 JavaScript 라이브러리입니다. **웹 화면을 레고 블록처럼 나누어 만드는 도구**라고 생각하면 됩니다.

React는 버튼, 메뉴, 카드, 목록 같은 화면 요소를 부품처럼 만들어 조립하는 방식으로 웹앱을 구성합니다. AI가 “React component”라는 말을 쓰면 웹 화면의 한 조각을 뜻하는 경우가 많습니다. Next.js는 React를 기반으로 웹사이트와 웹앱을 더 체계적으로 만들게 해주는 도구입니다.

React는 이런 화면을 만들 때 자주 보게 됩니다:

- 채널 포트폴리오 사이트
- 콘텐츠 캘린더 웹앱
- 내부 관리 페이지
- 랜딩페이지 UI

React는 화면 부품을 만드는 라이브러리입니다. Next.js는 React를 바탕으로 웹사이트·웹앱 전체 구조를 만드는 프레임워크입니다. HTML/CSS는 웹페이지의 구조와 모양을 담당합니다. **React는 ‘화면 조각을 만드는 도구’로 이해하면 충분합니다.**

### Next.js

Next.js는 웹사이트나 웹앱을 만들기 위한 React 기반 프레임워크입니다. 프레임워크는 앱을 정해진 구조로 만들게 도와주는 틀입니다. **웹사이트를 처음부터 맨손으로 짓는 대신, 방 배치와 기본 골격이 잡힌 건축 키트를 쓰는 것**과 비슷합니다.

Next.js는 페이지 구성, 라우팅, 서버 기능, 배포 구조 등을 함께 다루는 웹 개발 도구입니다. 엄밀히 말하면 개발 언어는 아니지만, 비개발자는 “AI가 웹앱을 만들 때 자주 쓰는 기술 이름”으로 이해하면 충분합니다.

AI가 이런 결과물을 만들 때 Next.js를 볼 수 있습니다:

- 채널 소개 사이트
- 강의 판매 페이지
- 콘텐츠 관리 대시보드
- 영상 아이디어 보드

Next.js는 프레임워크이고, React는 화면 부품을 만드는 라이브러리입니다. JavaScript/TypeScript는 Next.js 프로젝트에서 주로 쓰는 언어이고, Node.js는 개발 중 Next.js를 실행하는 환경으로 자주 쓰입니다. **Next.js를 언어가 아니라 웹앱을 짜는 큰 틀로 이해하면 됩니다.**

### HTML / CSS

HTML은 웹페이지의 구조를 만들고, CSS는 웹페이지의 모양을 꾸미는 기술입니다. **HTML이 집의 구조와 가구 배치라면, CSS는 벽지, 조명, 색감, 인테리어**입니다.

HTML은 제목, 문단, 이미지, 버튼 같은 웹페이지의 뼈대를 정합니다. CSS는 색상, 글꼴, 여백, 배치 같은 시각적 스타일을 정합니다. 엄밀히 말하면 Python 같은 프로그래밍 언어와는 다르지만, 웹 결과물을 이해할 때 가장 기본적으로 보게 되는 개발 기술 이름입니다.

HTML과 CSS는 이런 결과물을 만들 때 자주 등장합니다:

- 랜딩페이지
- 포트폴리오 페이지
- 영상 설명용 웹페이지
- 자막 미리보기 화면
- 뉴스레터 템플릿

HTML은 구조이고, CSS는 스타일입니다. JavaScript는 동작을 담당합니다. Next.js 같은 프레임워크는 이 기술들을 묶어 더 큰 웹앱을 만들게 해줍니다. **웹 결과물을 볼 때는 구조, 스타일, 동작을 나누어 보면 이해가 쉬워집니다.**
